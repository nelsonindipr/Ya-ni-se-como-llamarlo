Observations:

- Files under `docs/` currently exist but are empty.

## 2) What is missing

To build v0.1, the project needs:

- **Application skeleton** (frontend, backend, or monolith structure).
- **Package/dependency management** (for example `package.json`, `pyproject.toml`, etc.).
- **Domain model definitions** for teams, players, ratings, fixtures, games, standings.
- **Simulation engine** implementing possession/game flow aligned with BSN/FIBA context.
- **Data seed layer** with initial league, teams, and players.
- **UI flow** for matchup selection, simulation trigger, and box score/standings views.
- **Validation and tests** for deterministic simulation and standings calculations.

## 3) Tech stack currently in use

No active runtime stack is implemented yet.

Current state:
- **Language/framework:** Not defined.
- **Database:** Not defined.
- **Build tooling:** Not defined.
- **Testing framework:** Not defined.

Recommendation for v0.1 (fast delivery, maintainable):
- **Frontend:** React + TypeScript + Vite.
- **State/Data:** Local in-memory store (no DB yet).
- **Styling:** Simple CSS Modules or Tailwind (optional).
- **Testing:** Vitest (unit) + React Testing Library for core UI flow.

This keeps v0.1 lightweight while enabling later migration to API + DB.

## 5) Step-by-step plan to build Version 0.1

Scope includes only:
- teams
- players
- basic player ratings
- matchup selection
- simulate game button
- box score
- standings

Out of scope (explicitly deferred):
- contracts, trades, free agency, draft, injuries, playoffs

---

### Phase 0 — Foundation (Day 1)
1. Initialize React + TypeScript project with Vite.
2. Add lint/test tooling (ESLint + Vitest).
3. Create base folder structure and domain types.
4. Add README section for run/test commands.

**Deliverable:** App boots with placeholder pages and typed models.

### Phase 1 — Core domain models (Day 1–2)
1. Define `Team` model:
   - `id`, `name`, `city`, `abbrev`, `conference` (optional for future), `colors`.
2. Define `Player` model:
   - `id`, `teamId`, `name`, `position`, `age`.
3. Define `PlayerRatings` model (basic):
   - `insideScoring`, `midRangeScoring`, `threePointScoring`, `playmaking`, `perimeterDefense`, `interiorDefense`, `rebounding`, `stamina` (0–100).
4. Define `GameResult` and `BoxScore` models:
   - team totals + per-player line (`MIN`, `PTS`, `REB`, `AST`, `STL`, `BLK`, `TOV`, `FGM/FGA`, `3PM/3PA`, `FTM/FTA`).
5. Define `StandingRow` model:
   - `teamId`, `wins`, `losses`, `winPct`, `pointsFor`, `pointsAgainst`.

**Deliverable:** Static type-safe entities and sample seed data for 6–10 teams.

### Phase 2 — Seed data and ratings baseline (Day 2)
1. Add BSN/FIBA-inspired team set (fictional or real naming, depending licensing constraints).
2. Seed 10–12 players per team.
3. Build ratings profile templates by position:
   - PG higher playmaking; C higher interior defense/rebounding; etc.
4. Validate data (all players belong to existing teams; rating ranges enforced).

**Deliverable:** Complete in-memory league dataset.

### Phase 3 — Match engine v0.1 (Day 3–4)
1. Implement a **40-minute FIBA game model**:
   - 4 quarters × 10 minutes.
2. Simplified possession loop:
   - determine pace, possession count, shot zone choice (inside/mid/3), turnover chance, offensive rebound chance, free throw chance.
3. Team strength calculation from active players’ ratings.
4. Generate team score and per-player box score contribution using weighted sampling.
5. Guarantee data consistency:
   - team totals reconcile with player totals.
6. Add seeded RNG option for test reproducibility.

**Deliverable:** `simulateGame(homeTeamId, awayTeamId)` returns deterministic structure when seed is provided.

### Phase 4 — Gameplay UI flow (Day 4–5)
1. Build **Matchup Selection** screen:
   - dropdown/select for Home and Away teams.
2. Add **Simulate Game** button:
   - disabled until valid non-identical teams are selected.
3. Render **Box Score** view after simulation:
   - final score header + per-team player table.
4. Render **Standings** table:
   - update W/L after each simulation.
5. Add reset-season action (optional but useful for testing).

**Deliverable:** End-to-end playable v0.1 loop.

### Phase 5 — QA and balance pass (Day 5)
1. Unit tests:
   - rating weight calculations
   - possession outcomes in expected ranges
   - standings updates after results
2. Simulation sanity checks:
   - realistic final score ranges for FIBA-style pace.
3. UX checks:
   - empty/loading/error states.
4. Document known limitations and deferred features.

**Deliverable:** Stable v0.1 candidate with minimal automated coverage.

## Functional specification for v0.1 screens

1. **Home / Matchup Screen**
   - Team A selector
   - Team B selector
   - “Simulate Game” button
2. **Result Screen / Panel**
   - Final score
   - Quarter-by-quarter score (optional in v0.1, recommended)
   - Team box score tables
3. **Standings Screen / Section**
   - Team, W, L, Win%, PF, PA
   - Sorted by Win%, then point differential

## BSN/FIBA-specific design constraints for v0.1

- 40-minute game length (not NBA 48).
- Emphasize tactical half-court pace over high-possession NBA default.
- Keep rating impact balanced so team identity matters (defense/rebounding should visibly affect outcomes).
- Use slightly lower average scoring environment than NBA-like arcade simulations.

## Acceptance criteria for Version 0.1

- User can select two teams and simulate a game.
- Simulation produces final score + player box score.
- Standings update correctly after each game.
- Models and engine support FIBA-length game rules.
- No implementation of contracts/trades/free agency/draft/injuries/playoffs.
